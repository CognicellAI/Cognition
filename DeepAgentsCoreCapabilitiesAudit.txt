Areas Where Cognition Reimplements Deep Agents Capabilities
These are the problems to fix:
1. HIGH: Manual Event Parsing in astream_events() (runtime.py:367-413)
What it does: Calls self._agent.astream_events(version="v2") and manually parses raw LangGraph callback events (on_chat_model_stream, on_tool_start, on_tool_end, on_custom_event) into AgentEvent dataclasses.
What Deep Agents provides: The compiled graph supports astream() with stream_mode="messages" or stream_mode=["messages", "updates"] plus subgraphs=True for structured access to all events including subagent execution.
Concrete problems:
- Broken tool_call_id correlation (runtime.py:391,401): Uses f"{tool_name}_{id(data)}" — id(data) is a CPython memory address that differs between on_tool_start and on_tool_end events, so tool starts and results can never be correlated by the client
- No subagent visibility: Doesn't pass subgraphs=True, so all subagent execution is invisible to the streaming layer
- Fragile string matching: Depends on exact internal LangGraph event type strings
Plan: Replace with astream(stream_mode=["messages", "updates"], subgraphs=True, version="v2"). The AgentEvent types are fine to keep as Cognition's domain model, but the translation should use the structured v2 format, not raw callback events.
2. MEDIUM: ContextManager project indexing (context.py, cognition_agent.py:292-313)
What it does: At agent creation time, shells out to find . -type f | head -1000 then runs stat on each file (N+1 subprocess calls), runs naive regex scoring, and appends a static "Project Context" block to the system prompt.
What Deep Agents provides: ls, glob, grep tools that the agent can use dynamically during execution. MemoryMiddleware for AGENTS.md loading. The agent is capable of discovering project structure itself.
Concrete problems:
- N+1 subprocess calls during initialization (one stat per file up to 1000 files)
- Static snapshot — doesn't update as files change during the session
- Naive regex scoring adds minimal value vs. letting the agent explore dynamically
- Silently caught Exception (cognition_agent.py:308) — masked by fallback pattern (tracked as F-04 in ROADMAP)
Plan: Remove context.py entirely. If a project summary is desired, load it from AGENTS.md (which deep agents already supports via memory), or let the agent use its filesystem tools. The value-add does not justify the complexity.
3. LOW: Redundant write_todos instructions in system prompt
Two places:
- cognition_agent.py:189,198-203 — SYSTEM_PROMPT includes "Break down complex tasks using todos" and "First call write_todos..."
- deep_agent_service.py:692-700 — _build_messages() appends write_todos instructions if not already present
What Deep Agents provides: TodoListMiddleware injects the write_todos tool with its own description. BASE_AGENT_PROMPT in the deep agents graph already includes structured tool usage guidance. The todo list prompt section ("3. To-do list prompt") is part of Deep Agents' prompt assembly as documented in the harness capabilities page.
Plan: Remove the duplicate write_todos instructions from both locations. Keep only the Cognition persona portion of SYSTEM_PROMPT ("You are Cognition, an expert AI coding assistant..."). The deep agents harness handles tool usage guidance.
4. LOW: Dead tracking state in stream_response() (deep_agent_service.py:381-385)
_planning_mode = False
_plan_todos: list[dict[str, Any]] = []
_current_step_index = -1
_completed_steps: set[int] = set()
These variables are assigned but never read back anywhere meaningful — the stream_response() method just re-yields events without using this state. This is dead code from a planned feature that was never completed.
Plan: Remove the dead variables. If todo tracking is needed at the service layer, it should consume deep agents' native todo state from the graph state rather than manually tracking.
