Deep Agents Core Capabilities vs. Cognition: Alignment Analysis
Executive Summary
Cognition already has strong integration with deepagents via create_deep_agent(), but it's using the library more as a graph compiler than as a full harness. Several core Deep Agents capabilities are either partially utilized or not surfaced at all. Below is a capability-by-capability breakdown.
---
1. ALREADY ALIGNED (Using Well)
Virtual Filesystem / Backends
- Deep Agents: Pluggable backends (StateBackend, FilesystemBackend, LocalShellBackend, CompositeBackend, StoreBackend)
- Cognition: Uses FilesystemBackend, LocalShellBackend via sandbox_backend.py, and CompositeBackend to route /skills/api/ to ConfigRegistrySkillsBackend for DB-backed skills
- Status: Well aligned. Cognition extends the backend system appropriately for its multi-tenant needs.
Skills (Progressive Disclosure)
- Deep Agents: Directories with SKILL.md, progressive disclosure via SkillsMiddleware
- Cognition: Two skill sources (filesystem + DB-backed via ConfigRegistrySkillsBackend), API CRUD for skills, hot-reload via ConfigChangeDispatcher
- Status: Well aligned. Cognition goes beyond Deep Agents by adding API-managed skills with scope isolation.
Middleware
- Deep Agents: TodoListMiddleware, FilesystemMiddleware, SubAgentMiddleware, SummarizationMiddleware, AnthropicPromptCachingMiddleware, PatchToolCallsMiddleware
- Cognition: Adds ToolSecurityMiddleware, CognitionObservabilityMiddleware, CognitionStreamingMiddleware, plus resolution of upstream middleware (tool_retry, tool_call_limit, pii, human_in_the_loop) by name
- Status: Well aligned. Deep Agents middleware is applied internally by create_deep_agent(); Cognition layers its own on top.
Subagents
- Deep Agents: Dict-based and CompiledSubAgent definitions, general-purpose subagent auto-included
- Cognition: AgentDefinition.to_subagent() translates to Deep Agents SubAgent TypedDict, registry-managed definitions, YAML/Markdown agent files in .cognition/agents/
- Status: Well aligned. The AgentDefinitionRegistry adds API-level management that Deep Agents doesn't have.
Memory (AGENTS.md)
- Deep Agents: memory parameter loads AGENTS.md files at startup, always in system prompt
- Cognition: Passes memory paths from ConfigRegistry to create_deep_agent(), supports AGENTS.md
- Status: Aligned.
---
2. PARTIALLY ALIGNED (Gaps to Close)
Context Management / Summarization
- Deep Agents: Automatic offloading of large tool inputs/results (>20k tokens), automatic summarization at 85% of context window, configurable tool_token_limit_before_evict, optional SummarizationToolMiddleware for agent-triggered summarization
- Cognition: Relies on Deep Agents' built-in summarization passively, but does NOT expose configuration for tool_token_limit_before_evict and does NOT wire the optional SummarizationToolMiddleware
- Gap: Cognition should expose summarization configuration in config.yaml/ConfigRegistry and optionally enable the summarization tool middleware
Planning (Todo List)
- Deep Agents: Built-in write_todos tool, persisted in agent state, instructions in system prompt
- Cognition: Relies on Deep Agents' built-in todo system passively; no API to read/stream todo state to clients
- Gap: Cognition's SSE streaming layer (EventBuilder) doesn't have a dedicated todo_update event type. The client can't display a real-time todo list. Deep Agents' frontend docs (https://docs.langchain.com/oss/python/deepagents/frontend/todo-list.md) describe how to build this.
Human-in-the-Loop
- Deep Agents: interrupt_on parameter, per-tool approval (approve/edit/reject), subagent-level HITL, interrupt() primitive within tools, batched multi-tool interrupts
- Cognition: Passes interrupt_on from ConfigRegistry to create_deep_agent(), but the API layer (send_message route, SSE streaming) has no protocol for surfacing interrupts to the client and resuming with Command(resume=...). The HITL pipeline is configured but not end-to-end usable.
- Gap: This is a significant gap. The API needs interrupt event types in SSE, a resume endpoint (POST /sessions/{id}/resume), and client-side handling.
Structured Output
- Deep Agents: response_format parameter on create_deep_agent() for Pydantic model validation
- Cognition: Does NOT pass response_format to create_deep_agent(). No API or config path to specify structured output schemas.
- Gap: For use cases like evaluation pipelines or structured API responses, this should be wired through.
Streaming
- Deep Agents: v2 streaming format, subgraph streaming with namespaces, stream_mode=["updates", "messages", "custom"], get_stream_writer() for custom events
- Cognition: DeepAgentRuntime.astream_events() uses the older astream_events(version="v2") API (LangGraph v2 events like on_chat_model_stream), NOT the newer stream(version="v2") format. It manually translates events into canonical types (TokenEvent, ToolCallEvent, etc.). Subagent streaming is partially handled via DelegationEvent but not as richly as the new v2 format enables.
- Gap: Cognition could benefit from migrating to the newer stream() API with subgraphs=True for better subagent lifecycle tracking.
---
3. NOT INCORPORATED (New Opportunities)
Long-Term Memory (Cross-Thread Persistence)
- Deep Agents: CompositeBackend routing /memories/ to StoreBackend backed by LangGraph Store (Postgres/InMemory); agents can read/write persistent files across threads; self-improving instructions pattern
- Cognition: Has no cross-thread memory. ConfigRegistry stores per-entity config but not agent working memory. Each session is isolated.
- Opportunity: This directly supports Cognition's mission. A StoreBackend route for /memories/ could be added alongside the existing CompositeBackend that routes /skills/api/. This is P3 roadmap material — it requires deciding on the store backend (Postgres store is natural given Cognition already supports Postgres).
Agent Client Protocol (ACP)
- Deep Agents: deepagents-acp package exposes agents over ACP for IDE integration (Zed, JetBrains, VSCode, Neovim)
- Cognition: Has its own REST+SSE API but no ACP support
- Opportunity: ACP integration would let Cognition agents be used directly from code editors. This is an additive feature that doesn't conflict with the existing API. Worth adding as a P3 feature — separate transport layer alongside REST+SSE.
Connection Resilience (Model Retry Config)
- Deep Agents: max_retries and timeout configurable on init_chat_model(), with exponential backoff for 429/5xx
- Cognition: _build_model() calls init_chat_model() but does NOT expose max_retries or timeout as configurable parameters
- Opportunity: Simple win. Expose max_retries and timeout in ProviderConfig or config.yaml. Low effort, high value for production reliability.
Summarization Tool Middleware
- Deep Agents: create_summarization_tool_middleware() gives the agent a tool to trigger summarization proactively (e.g., between tasks)
- Cognition: Not wired
- Opportunity: Add as an optional middleware that can be enabled via config.yaml under agent.middleware. Low effort.
Frontend Todo/Subagent Streaming
- Deep Agents: Dedicated frontend patterns for real-time todo list display and subagent progress cards
- Cognition: SSE EventBuilder has token, tool_call, tool_result, planning, step_complete, delegation, usage, status, error, done — but no todo_update event
- Opportunity: Add a todo_update SSE event type that surfaces the agent's write_todos state changes. This makes the todo list a first-class streaming primitive. Medium effort (needs state diffing).
---
4. Recommended Incorporation Plan (Mission-Aligned)
Here's what I'd recommend adding to the ROADMAP, organized by effort and impact:
Quick Wins (P1-level, <1 day each)
1. Expose max_retries / timeout in ProviderConfig — Wire through _build_model(). Production resilience.
2. Expose tool_token_limit_before_evict in config.yaml — Pass to create_deep_agent(). Context management tuning.
3. Wire SummarizationToolMiddleware — Add as a named middleware option alongside tool_retry, tool_call_limit, etc.
Medium Effort (P2-level, 1-3 days each)
4. Todo streaming events — Add todo_update to EventBuilder and SSEStream. Extract todo state from agent graph state changes. Surfaces planning to clients.
5. Structured output support — Add response_format to AgentDefinition and SessionConfig. Pass through to create_deep_agent(). Return validated response in message metadata.
6. Human-in-the-Loop end-to-end — Add interrupt SSE event type, POST /sessions/{id}/resume endpoint, and Command(resume=...) wiring. This unlocks the interrupt_on config that's already being passed.
Larger Features (P3-level, 3-5 days each)
7. Long-term memory via StoreBackend — Add /memories/ route to the CompositeBackend. Requires choosing a BaseStore implementation (likely PostgresStore). Cross-thread persistence for preferences, knowledge, research.
8. ACP transport layer — Add deepagents-acp as optional dependency. Expose the same agent through ACP alongside REST+SSE. IDE integration.
